## Part 1 — Instance startup stages

Oracle has 3 startup stages. PostgreSQL has an equivalent for each.

| Oracle stage | What happens | PostgreSQL equivalent |
|---|---|---|
| NOMOUNT | Reads init params, starts SGA | `pg_ctl start` — reads `postgresql.conf` |
| MOUNT | Reads control file, not open to users | Internal — mounts data directory |
| OPEN | Database open, users can connect | Fully started, accepting connections |

### Lab steps

Check your instance is running and see its PID (equivalent of Oracle's instance identifier):

```bash
sudo cat /var/lib/postgresql/16/main/postmaster.pid
```

The first line is the PID. As long as this file exists, the instance is running — when PostgreSQL stops, this file is deleted. This is exactly how Oracle uses its instance identifier.

Simulate a controlled shutdown and restart:

```bash
sudo systemctl stop postgresql
sudo ls /var/lib/postgresql/16/main/postmaster.pid
# File should be gone — instance is down

sudo systemctl start postgresql
sudo cat /var/lib/postgresql/16/main/postmaster.pid
# File is back — instance is up
```

### ✅ Results observed

```
# After start:
3093
/var/lib/postgresql/16/main
1777699834
5432
/var/run/postgresql
localhost
   918459         1
ready
```

- Line 1 (`3093`) = master process PID
- Line 4 (`5432`) = port
- Last line (`ready`) = instance is in OPEN stage, accepting connections
- File disappeared on stop, reappeared on start — confirmed

## Part 2 — Initialization parameters (equivalent of init.ora / SPFILE)

In Oracle, `init.ora` and SPFILE store instance parameters. PostgreSQL uses `postgresql.conf` and `postgresql.auto.conf` (changed via `ALTER SYSTEM`).

```bash
# View the main config file (equivalent of init.ora):
sudo nano /etc/postgresql/16/main/postgresql.conf
```

Key parameters to find and note down:

```
max_connections        # Oracle: SESSIONS parameter
shared_buffers         # Oracle: DB_CACHE_SIZE
wal_level              # Oracle: LOG_MODE (archivelog/noarchivelog)
archive_mode           # Oracle: ARCHIVE_LOG_START
listen_addresses       # Oracle: LOCAL_LISTENER
```

Query them live (equivalent of Oracle's `SHOW PARAMETER`):

```sql
sudo -u postgres psql -c "
SELECT name, setting, unit, context
FROM pg_settings
WHERE name IN (
  'max_connections',
  'shared_buffers',
  'wal_level',
  'archive_mode',
  'listen_addresses',
  'log_destination'
)
ORDER BY name;"
```

Change a parameter dynamically (equivalent of Oracle's `ALTER SYSTEM`):

```sql
sudo -u postgres psql -c "ALTER SYSTEM SET work_mem = '8MB';"
sudo -u postgres psql -c "SELECT pg_reload_conf();"

-- Verify it changed (written to postgresql.auto.conf):
sudo -u postgres psql -c "SHOW work_mem;"
```

### ✅ Results observed

```
       name       |  setting  | unit |  context
------------------+-----------+------+------------
 archive_mode     | off       |      | postmaster
 listen_addresses | localhost |      | postmaster
 log_destination  | stderr    |      | sighup
 max_connections  | 100       |      | postmaster
 shared_buffers   | 16384     | 8kB  | postmaster
 wal_level        | replica   |      | postmaster
```

| Parameter | Value | Meaning |
|---|---|---|
| `archive_mode` | off | WAL archiving disabled — equivalent to Oracle NOARCHIVELOG mode |
| `listen_addresses` | localhost | Only local connections accepted — consistent with security hardening done in Lab Session |
| `shared_buffers` | 16384 × 8kB = 128MB | Database buffer cache size |
| `wal_level` | replica | Enough WAL for replication but not full logical decoding |
| `max_connections` | 100 | Maximum concurrent sessions |

**Parameters with context `postmaster`** require a full restart to take effect — they are allocated at startup when shared memory is set. Parameters with context `sighup` can be reloaded live with `pg_reload_conf()`.

After `ALTER SYSTEM SET work_mem = '8MB'` and `pg_reload_conf()`:

```
# postgresql.auto.conf contents:
# Do not edit this file manually!
# It will be overwritten by the ALTER SYSTEM command.
work_mem = '8MB'
```

`postgresql.auto.conf` always wins over `postgresql.conf` when both define the same parameter — same as Oracle SPFILE superseding `init.ora`.

## Part 3 — The alert log

Oracle's alert log records all major instance events. PostgreSQL equivalent is in `/var/log/postgresql/`.

```bash
# Find your log file:
ls /var/log/postgresql/

# Tail the log (like Oracle's alert_<SID>.log):
sudo tail -50 /var/log/postgresql/postgresql-16-main.log
```

Look for lines showing:
- Database system was shut down / started up
- Checkpoint activity
- Any errors or warnings

Enable more verbose logging to see more Oracle-like detail:

```bash
sudo -u postgres psql -c "ALTER SYSTEM SET log_checkpoints = on;"
sudo -u postgres psql -c "ALTER SYSTEM SET log_connections = on;"
sudo -u postgres psql -c "ALTER SYSTEM SET log_disconnections = on;"
sudo -u postgres psql -c "SELECT pg_reload_conf();"

# Now restart and watch the log:
sudo systemctl restart postgresql
sudo tail -30 /var/log/postgresql/postgresql-16-main.log
```

## Part 4 — Dynamic performance views (V$ equivalent)

Oracle's V$ views expose live instance data. PostgreSQL has `pg_stat_*` views.

```sql
sudo -u postgres psql

-- V$SESSION equivalent:
SELECT pid, usename, application_name, client_addr, state, query
FROM pg_stat_activity;

-- V$SYSSTAT equivalent (database-wide stats):
SELECT datname, numbackends, xact_commit, xact_rollback,
       blks_read, blks_hit, tup_inserted, tup_updated, tup_deleted
FROM pg_stat_database
WHERE datname = 'postgres';

-- V$BGPROCESS equivalent:
SELECT pid, backend_type, wait_event_type, wait_event
FROM pg_stat_activity
WHERE backend_type != 'client backend';

-- Calculate buffer cache hit ratio (key DBA metric):
SELECT
  round(blks_hit * 100.0 / (blks_hit + blks_read), 2) AS cache_hit_ratio
FROM pg_stat_database
WHERE datname = 'postgres';
```

A cache hit ratio above 95% is healthy. Below 90% means `shared_buffers` may need increasing.

## Key takeaway

Oracle Enterprise Manager and SQL*Plus = PostgreSQL's `pg_stat_*` views + `psql` + log files. The data is the same — startup events, session info, parameter values, performance counters. The tools just look different.

## Scenario: "The database was restarting by itself over the weekend"

**Step 1 — Check current status**

```bash
sudo systemctl status postgresql
```

Look for `Active: active (running)`. If it shows `failed` or `inactive`, the instance is currently down. `systemctl status` also shows the last few log lines inline, giving an immediate first clue.

**Step 2 — Read the alert log for the weekend window**

```bash
sudo grep "2026-05-02\|2026-05-03" /var/log/postgresql/postgresql-16-main.log
# Adjust dates to match the weekend in question
```

Or just tail a larger window:

```bash
sudo tail -200 /var/log/postgresql/postgresql-16-main.log
```

You are looking for two specific patterns:

**Pattern A — repeated restarts (confirms the claim):**
```
LOG:  database system is shut down
LOG:  starting PostgreSQL ...
LOG:  database system is shut down
LOG:  starting PostgreSQL ...
```
Multiple shutdown/startup cycles in a short window means it was restarting repeatedly.

**Pattern B — crash vs clean shutdown (tells you why):**

If each shutdown shows:
```
LOG:  database system was interrupted
LOG:  automatic recovery in progress
LOG:  redo starts at ...
LOG:  redo done at ...
```
→ The database was being killed hard, not shut down cleanly. Something external was terminating the process — OOM killer, host reboot, or a monitoring script.

If each shutdown shows:
```
LOG:  received fast shutdown request
LOG:  database system is shut down
```
→ Someone or something was issuing a legitimate stop command.

**Step 3 — Verify the current restart was clean**

The most recent startup entry in the log should show:
```
LOG:  database system was shut down at <timestamp>
LOG:  database system is ready to accept connections
```
No `automatic recovery in progress` = the last startup was clean. If recovery lines are present, the database may still be catching up.

**Step 4 — Confirm all background processes are alive**

```bash
sudo -u postgres psql -c "
SELECT pid, backend_type, wait_event
FROM pg_stat_activity
WHERE backend_type != 'client backend';"
```

You should see all 5 processes: checkpointer, background writer, walwriter, autovacuum launcher, logical replication launcher. Any missing means the instance is not fully healthy.

**Step 5 — Check cache hit ratio for performance impact**

```bash
sudo -u postgres psql -c "
SELECT round(blks_hit * 100.0 / (blks_hit + blks_read), 2) AS cache_hit_ratio
FROM pg_stat_database
WHERE datname = 'postgres';"
```

Above 95% = healthy. Below 90% = the restarts may have caused cache to be cold and performance is degraded. If degraded, it will recover naturally as the buffer cache warms up — no action needed unless it stays low.

**What a healthy outcome looks like:**
- `systemctl status` shows `active (running)`
- Log shows one clean startup, no repeated restart pattern recently
- Most recent startup has no recovery lines
- All 5 background processes present
- Cache hit ratio above 95%

**What an unhealthy outcome looks like:**
- Log shows crash recovery on the most recent startup → data integrity risk, investigate further
- Fewer than 5 background processes → instance is partially started
- Cache hit ratio below 90% → performance is degraded, monitor and give it time to warm up
- Repeated restart pattern still continuing → there is an active problem, do not leave it unattended
